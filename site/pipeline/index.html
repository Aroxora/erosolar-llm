Recommended smallest base

Qwen3-0.6B
Qwen/Qwen3-0.6B · 0.6B params (dense) · 32,768 ctx
Apache-2.0 confirmed general-purpose dense — simplest to QLoRA & serve

This is the smallest model in the verified set that clears both gates that matter for erosolar: a permissive, commercial-safe license (Apache-2.0, OSI-open) and a confirmed genuinely general-purpose assistant. Everything smaller fails one gate — Gemma 3 270M and SmolLM2-135M are not usable general assistants; the ~350M tier that is general-purpose is either non-OSI-licensed (LiquidAI LFM2-350M) or only marginally smaller (IBM Granite 4.0-350M, SmolLM2-360M).

Expected performance: entry / sub-1B tier — basic instruction-following, short-form chat, and light tool-calling as a proto/dial-in model. Not a frontier-quality assistant. No independent benchmark scores were cited during research, so we report the capability tier qualitatively and refuse to invent numbers. Treat it as the cheapest rung to validate the fine-tune pipeline before scaling up.

It ships in the repo: erosolar_agent/finetune/configs/qwen3-0.6b-floor.yaml is the rung-1 config, and TrainConfig.base_model now defaults to unsloth/Qwen3-0.6B.

MoE fine-tune gotcha: for the MoE rungs (7, 10, 11) QLoRA must load all total params (30.5B / 35B / 25.2B) into 4-bit VRAM even though only ~3–4B activate per token. Fine-tune VRAM tracks total params, not the small active count — the active-param saving is an inference benefit.

Fine-tune cost by size (QLoRA / LoRA)

Primary source: the official Unsloth requirements table (fetched 2026-06-14), which Unsloth states are “the absolute minimum” and notes that “sometimes more VRAM is required depending on the model.” QLoRA = LoRA adapters over a 4-bit-quantized frozen base; 4-bit quant alone saves ~75% memory vs 16-bit.

Verified rows — QLoRA 4-bit min / LoRA 16-bit min

  • 3B = 3.5 GB / 8 GB · 7B = 5 / 19 · 8B = 6 / 22 · 9B = 6.5 / 24
  • 11B = 7.5 / 29 · 14B = 8.5 / 33 · 27B = 22 / 64 · 32B = 26 / 76
  • 70B = 41 / 164 · 405B = 237 / 950

Estimated (not in Unsloth's table — labeled as estimates)

  • 0.5B ≈ 2–3 GB QLoRA · 1B ≈ 3 GB (Unsloth advertises fine-tuning “with just 3 GB VRAM”) · 4B ≈ 4 GB (interpolated 3B↔7B).
  • So rungs 1–4 fit a free Colab/Kaggle T4 (16 GB) trivially; rung 5 (8B) fits 8 GB at QLoRA but wants a 24 GB card for 16-bit LoRA; rung 6 (14B) is comfortable QLoRA on a single 24 GB 4090; rung 8 (32B) at 26 GB just exceeds a 24 GB 4090, so a 40 GB+ card (A100/H100) is the safe pick.

Unsloth's role (verified): QLoRA + hand-written Triton kernels — per its Llama-3.3 blog, ~2× faster and ~70% less memory than Flash-Attention-2 + Hugging Face with mathematically identical results (up to 80% reduction cited for vision RL). Minimum GPU: CUDA capability 7.0+ (V100, T4, RTX 20/30/40/50, A100, H100).

Foundational claim (verified — QLoRA paper, Dettmers et al., arXiv:2305.14314): QLoRA “reduces memory usage enough to finetune a 65B parameter model on a single 48 GB GPU while preserving full 16-bit finetuning task performance.”

DPO is treated as roughly equal to SFT at the same precision; the exact overhead depends on whether a separate reference model is loaded (TRL/Unsloth can share or skip it). All figures are minimums at batch size 1–2 and modest sequence length — longer context, bigger batches, or disabling gradient checkpointing raise them.

Caveats & honesty notes

  • No benchmark numbers. None were cited or fetched for any ladder model during research, so every “expected performance” is a qualitative tier with zero numeric scores. We will not add MMLU/GPQA/etc. figures here without a fetched primary source.
  • License gotchas — not all “open” is OSI-open. Gemma 3 (270M–27B) uses Google's custom Gemma license (accept-terms; not OSI); note that Gemma 4 switched to Apache-2.0. Llama 3.2 / Llama 4 use Meta Community licenses (700M-MAU clause, acceptable-use, jurisdictional limits). LiquidAI LFM2 uses a vendor license with a sub-$10M-revenue gate. Cleanly permissive in this set: Qwen3 / Qwen3.5 (Apache-2.0), Phi-4-mini (MIT), IBM Granite 4.x (Apache-2.0), SmolLM2 (Apache-2.0), Mistral Small 3.1/3.2 (Apache-2.0), and Gemma 4 (Apache-2.0).
  • Modality gotcha. Qwen3.5 small + MoE variants (rungs 9, 10) are vision-language models, not text-only — pick them only if multimodality is wanted; a text-only fine-tune still loads (but ignores) the vision encoder.
  • Post-cutoff uncertainty. Qwen3.5 (~2026-02/03) and Gemma 4 (~2026-03/04) are recent; some facts come partly from Hugging Face metadata, so treat exact release dates as approximate. Gemma 4 26B-A4B is marked likely, not confirmed.
  • VRAM estimates. Sub-3B and all MoE figures are extrapolations; only 3B/7B/8B/9B/11B/14B/27B/32B are verified-from-table.
  • Param-count rounding. “NB” names round: Qwen3-8B is 8.2B, Qwen3-14B is 14.8B, Qwen3-32B is 32.8B total.
